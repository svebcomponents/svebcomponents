The recommended starting point is the official [template repo](https://github.com/svebcomponents/template):

```bash
pnpx degit svebcomponents/template my-project
cd my-project
pnpm install
pnpm dev
```

The template gives you a small pnpm workspace:

- `components/example-component`: a Svelte custom element package
- `apps/svelte-kit`: a SvelteKit app that consumes the custom element
- `configs/*`: shared linting, formatting, and TypeScript config packages

## Example Component

Let's take a look at the example component: `components/example-component/src/ExampleComponent.svelte`.

```svelte
<script lang="ts">
  let {
    increments = 1,
  }: {
    increments: number;
  } = $props();

  let count = $state(0);
</script>

<button onclick={() => (count += increments)}>
  add {increments} to {count}
</button>
```

If you know svelte 5, there is literally nothing unfamiliar in here.

```ts
increments: number;
```

The type annotation is enough information for `@svebcomponents/auto-options` to generate the custom element prop metadata Svelte needs later, when the `increments` HTML attributes has to be converted from string to number.

## Component Entrypoint

The package entrypoint imports the Svelte component and registers the generated custom element constructor:

```ts
import ExampleComponent from "./ExampleComponent.svelte";

export default ExampleComponent;

if (!customElements.get("example-component") && "element" in ExampleComponent) {
  customElements.define(
    "example-component",
    ExampleComponent.element as CustomElementConstructor,
  );
}
```
> [!INFO]
> When Svelte compiles a component as a custom element, it exposes the constructor as `Component.element`.

> [!NOTE]
> Usually svelte immediately registers the component with the custom element registry, but `svebcomponents` makes this explicit allowing you to tailor registration to your needs.

## Package Exports

The `exports` field of your component's `package.json` tells consumers  where the web component, and, if you want to opt into the experimental SSR path, the renderer live.

```json
{
  "exports": {
    ".": {
      "import": "./dist/client/index.js"
    },
    "./ssr": {
      "import": "./dist/server/ssr.js"
    }
  }
}
```

This declaration doubles as your config, as the `svebcomponents` CLI reads it to infer which components it has to build

## Using the Component

You use the svelte generated web component just as you would use any other web component:

```svelte
<script lang="ts">
  import "@svebcomponents/example-component";
</script>

<example-component increments={5}></example-component>
```

> [!WARNING]
> One footgun here worth calling out.
> Svelte 5 sets attributes as JavaScript properties under certain conditions.
> Therefore, you should use JavaScript expressions for non-string values when consuming custom elements from Svelte.
> In the example above, `increments={5}` passes the number `5`, where writing `increments="5"` passes the string `"5"` from Svelte's JavaScript side, which bypasses the custom element attribute converter.
> see: https://github.com/sveltejs/svelte/issues/12624

## SSR Setup

The template also showcases the experimental SSR path, because that is part of where this project is trying to go.

> [!DANGER]
> Please do not use SSR in production as it is still in an experimental stage


In `apps/svelte-kit/vite.config.ts`, the SvelteKit adds the Svebcomponents Vite transform plugin:

```ts
import { defineConfig } from "vite";
import { sveltekit } from "@sveltejs/kit/vite";
import svebcomponents from "@svebcomponents/ssr/vite";

export default defineConfig({
  plugins: [svebcomponents(), sveltekit()],
});
```

In `apps/svelte-kit/src/hooks.server.ts`, it registers the component renderer:

```ts
import { ElementRendererRegistry } from "@svebcomponents/ssr";
import ExampleComponentRenderer from "@svebcomponents/example-component/ssr";
import type { Handle } from "@sveltejs/kit";

ElementRendererRegistry.set("example-component", ExampleComponentRenderer);

export const handle: Handle = async ({ event, resolve }) => {
  return await resolve(event);
};
```

This registration step is telling the server: "when you see `<example-component>`, this is the renderer that knows what to do with it."
