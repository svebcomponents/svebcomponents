## Package Entrypoints

`@svebcomponents/build` uses the `exports` field of your component's `package.json` to allow for no-config builds.
For this to work, the CLI makes a few assumptions about how your project is structured. If you need more flexibility, you can always fall back to a manual configuration.

### Client Entrypoint (Required)

The client entrypoint is the module consumers import for the side effect of defining the custom element with the custom element registry:

```ts
import Component from "./Component.svelte";

if (!customElements.get("favorite-number") && Component.element) {
  customElements.define("favorite-number", Component.element);
}

export default Component;
```

`@svebcomponents/build` tries to infer this from your package exports.
If an export points at `dist/client/*`, the CLI maps it back to the matching source file under `src/*`.

```json
{
  "exports": {
    ".": {
      "import": "./dist/client/index.js"
    }
  }
}
```

That export tells the CLI: build `src/index.ts` into `dist/client/index.js`.

### Server Entrypoint (Optional)

For SSR, Svebcomponents follows Lit's `ElementRenderer` model. For this, component packages can expose server renderers at `./ssr`

```json
{
  "exports": {
    ".": {
      "import": "./dist/client/index.js"
    },
    "./ssr": {
      "import": "./dist/server/ssr.js"
    }
  }
}
```

## auto-options

In Svelte 5, props usually live in `$props()`.

```svelte
<script lang="ts">
  interface Props {
    label: string;
    value: number;
  }

  let { label, value }: Props = $props();
</script>
```

`@svebcomponents/auto-options` reads that declaration before the Svelte compiler runs. It uses the prop names and TypeScript types to generate the custom element options you would otherwise have to write by hand.

## Attribute Names and Prop Names

HTML wants `kebab-case`. JavaScript wants `camelCase`.

Those worlds meet at the custom element boundary, so Svebcomponents kebab-cases prop names for attributes:

| Prop name        | Attribute name    |
| ---------------- | ----------------- |
| `value`          | `value`           |
| `favoriteNumber` | `favorite-number` |
| `isFeatured`     | `is-featured`     |

Your component keeps normal JavaScript names. Consumers get HTML-shaped markup. Everybody can pretend this was obvious all along.

## Attribute Conversion

HTML attributes are strings. Always. Even when they look like numbers and booleans and other small lies.

Svelte custom elements need prop metadata to convert those strings into richer JavaScript values.

```svelte
<favorite-number value="42" featured="true"></favorite-number>
```

With inferred metadata, Svelte can treat those attributes as typed props:

- `value` can become a number
- `featured` can become a boolean
- object and array props can use Svelte's custom element conversion behavior

Without the metadata, attributes stay string-shaped.

## Svelte Consumers and Non-string Values

Here is the part that tends to surprise people.

When a Svelte app uses a custom element, values written as JavaScript expressions are assigned as properties:

```svelte
<favorite-number value={42}></favorite-number>
```

That passes the number `42`. Great.

Values written as quoted markup are strings:

```svelte
<favorite-number value="42"></favorite-number>
```

In browser-authored HTML, custom element attribute converters can turn string attributes into typed props. In Svelte-authored markup, Svelte may set the JavaScript property directly when it can see that the property exists.

If that direct property assignment receives the string `"42"`, the custom element attribute converter is not involved. The converter is standing right there, but Svelte has already walked past it.

Use JavaScript expressions for non-string values when consuming custom elements from Svelte.

## Manual Overrides

Inference should save you from boilerplate, not trap you inside it.

You can write custom element prop options manually when you need a different attribute name, type, or reflection behavior.

```svelte
<svelte:options
  customElement={{
    props: {
      value: { type: "Number", attribute: "data-value", reflect: false },
    },
  }}
/>
```

`auto-options` preserves the fields you wrote manually and fills in missing information where it can.
